What makes this system different (not an LLM)

This repo implements a deterministic, measurement‑first reasoning loop. It is designed to be auditable and reproducible, not a probabilistic text generator.

  • Explicit measurements: scores, constraints, and objective alignment are computed before decisions.
  • Relational substrate: semantic + 3D data attach to the same measured structure.
  • Auditable outputs: decision signals, rationale, and schedules are written to files.
  • Determinism‑aware: fixed timestamps + eval gates protect correctness.

How to use it

This is the fastest way to get the desktop Copilot app (or any external reviewer) enough code context to produce a grounded assessment.

  1. Build a safe, allowlist-based mirror (choose a profile).
  2. Export attachment-friendly copies (exports follow public_mirror/mirror_manifest.json).
  3. Option A (recommended): publish public_mirror/ as a temporary public repo.
  4. Option B (fallback): attach the exported *.txt files directly in Copilot app.
py -3 scripts/create_public_mirror.py
py -3 scripts/create_public_mirror.py --profile core_thinking
py -3 scripts/export_copilot_app_attachments.py
explorer Copilot_app_Attachments_txt_files_of_py_modules\exports
Which profile should I use?
  • adversarial: smallest mirror; fastest to publish/review; focuses on the adversarial harness and core deps.
  • core_thinking: larger mirror for deeper code review; includes module_*.py plus selected docs.

Both are allowlist-based and intentionally exclude runtime folders like LongTermStore/ and TemporaryQueue/.

Staging Observation Window (what it is)

This is a short, repeatable runbook to validate that the integrated AI Brain stack behaves as expected in realistic runs: adversarial harness behavior, adaptive sampling behavior, and rollback / escalation policy behavior.

How it’s part of the AI Brain

The “AI Brain” here is the repo’s end-to-end cycle (storage → retrieval → reasoning → selection → verification/error-resolution → scheduling) surfaced through cli.py and validated by run_eval.py.

The observation window runs the same integrated code paths you rely on for real operation, but with deterministic knobs available so you can reproduce and debug regressions.

How it pertains to metrics data

During runs, modules can emit counters and values through module_metrics.py, which are flushed to TemporaryQueue/metrics.json (repo convention). Example: rollback-storm escalations increment resolution_rollback_storm_escalations_total in module_error_resolution.py.

The command metrics_dashboard.py (wrapper → scripts/metrics_dashboard.py) summarizes those metrics so you can track drift (escalations, sampling behavior, etc.) across repeated staging cycles.

What artifacts to keep

For a staging window you typically keep: TemporaryQueue/metrics.json, pytest_adversarial.out, run_eval.out, and (if you run sweeps) TemporaryQueue/adversarial_sweep/sweep_results.csv plus the per-run adversarial_report_*.json files.
